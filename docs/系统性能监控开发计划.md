# 系统性能监控开发计划

## 1. 文档目标

本文档用于规划 RecipeRec 菜谱推荐系统中“系统性能监控”部分的开发工作。该部分的目标是在不影响现有推荐、搜索、反馈、登录、前端展示等功能的前提下，为后端服务增加基础可观测能力，方便后续定位接口耗时、资源瓶颈、推荐链路缓存命中与兜底情况。

本计划只覆盖系统性能监控开发，不直接改造 MapReduce 多模型并发推理逻辑；但会预留多模型推理指标接口，供后续并发优化模块接入。

## 2. 当前项目现状

当前项目已经具备一个基础 `/metrics` 接口：

- 位置：`api/routes.py`
- 调用：`monitor/metrics.py` 中的 `build_prometheus_metrics`
- 现有能力：输出服务存活、运行时长、Redis 状态、FAISS 索引规模、反馈总数、曝光数、点击数等简单统计。

当前监控能力的不足：

- 缺少 CPU、内存、磁盘、网络、进程资源指标。
- 缺少统一 FastAPI HTTP middleware，无法自动统计所有接口的请求量、耗时、状态码和异常数。
- 推荐接口的缓存命中、离线 CSV 命中、空结果、兜底等业务指标还没有完整埋点。
- 多模型推理耗时、成功、失败、超时等指标还没有统一封装。
- Prometheus 和 Grafana 配置尚未接入 `docker-compose.yml`。
- 现有 `/metrics` 通过手工拼接字符串生成指标，后续扩展成本较高。

## 3. 开发原则

为保证“不影响系统其他功能”，本次监控开发遵循以下原则：

| 原则 | 说明 |
| --- | --- |
| 旁路采集 | 监控逻辑只记录指标，不参与推荐结果计算。 |
| 不改响应结构 | 不修改现有 API 的 response model、字段名、状态码和业务返回语义。 |
| 不阻塞主链路 | 指标采集失败时只记录日志，不抛出影响用户请求的异常。 |
| 可关闭 | 通过配置项控制监控是否启用，必要时可以一键关闭。 |
| 低基数标签 | Prometheus 标签中不使用 `user_id`、`recipe_id`、`request_id` 等高基数字段。 |
| 保持兼容 | 保留已有 `/metrics` 路径和已有 `reciperec_*`、`movierec_*` 指标别名，避免旧看板失效。 |
| 小步接入 | 优先接入系统资源和 HTTP 指标，再逐步补充推荐业务指标。 |

## 4. 功能范围

### 4.1 必须完成

- 系统资源监控：CPU、内存、磁盘、网络、进程 CPU、进程内存。
- FastAPI 接口监控：请求总数、响应耗时、HTTP 状态码、异常数。
- 推荐接口业务监控：推荐请求数、返回结果数量、空结果、缓存命中、CSV 命中、兜底次数。
- 反馈接口业务监控：曝光、点击、收藏、评分、负反馈等事件数量。
- `/metrics` 输出 Prometheus 文本格式指标。
- Prometheus 抓取配置。
- Grafana 看板配置或看板说明。
- 基础测试与验收说明。

### 4.2 预留但不强依赖

- 多模型推理指标：模型耗时、成功、失败、超时、返回数量。
- Reduce 融合指标：输入数量、输出数量、融合耗时。

这些指标先在监控模块中定义好函数，不要求本阶段完成并发推理服务接入。

### 4.3 不在本次范围

- 不重写推荐算法。
- 不重构前端页面。
- 不改动用户登录权限逻辑。
- 不改动数据库表结构。
- 不把离线 Spark、ALS、ItemCF、LightGCN 训练流程纳入监控改造。
- 不实现完整告警系统，只预留 Prometheus/Grafana 展示能力。

## 6. 指标设计

### 6.1 系统资源指标

| 指标名 | 类型 | 标签 | 含义 |
| --- | --- | --- | --- |
| `system_cpu_usage_percent` | Gauge | 无 | 整机 CPU 使用率。 |
| `system_cpu_count` | Gauge | 无 | CPU 核心数。 |
| `process_cpu_usage_percent` | Gauge | 无 | FastAPI 进程 CPU 使用率。 |
| `system_memory_usage_percent` | Gauge | 无 | 系统内存使用率。 |
| `system_memory_available_bytes` | Gauge | 无 | 系统可用内存。 |
| `process_memory_rss_bytes` | Gauge | 无 | FastAPI 进程 RSS 内存。 |
| `system_disk_usage_percent` | Gauge | `mount` | 磁盘使用率。 |
| `system_disk_free_bytes` | Gauge | `mount` | 磁盘剩余空间。 |
| `system_disk_read_bytes_total` | Counter | 无 | 累计磁盘读取字节数。 |
| `system_disk_write_bytes_total` | Counter | 无 | 累计磁盘写入字节数。 |
| `system_network_sent_bytes_total` | Counter | 无 | 累计网络发送字节数。 |
| `system_network_recv_bytes_total` | Counter | 无 | 累计网络接收字节数。 |

### 6.2 HTTP 接口指标

| 指标名 | 类型 | 标签 | 含义 |
| --- | --- | --- | --- |
| `http_requests_total` | Counter | `method`, `path` | 接口请求总数。 |
| `http_request_duration_seconds` | Histogram | `method`, `path` | 接口响应耗时。 |
| `http_response_status_total` | Counter | `method`, `path`, `status_code` | HTTP 状态码数量。 |
| `http_exceptions_total` | Counter | `method`, `path`, `exception_type` | 接口异常数量。 |

标签说明：

- `path` 使用路由模板，例如 `/recipes/recommend/{user_id}`，不要使用真实用户 ID。
- `status_code` 只记录状态码，例如 `200`、`404`、`500`。
- 不记录请求体、用户 ID、菜谱 ID、token 等敏感或高基数字段。

### 6.3 推荐业务指标

| 指标名 | 类型 | 标签 | 含义 |
| --- | --- | --- | --- |
| `recommend_requests_total` | Counter | 无 | 推荐接口调用次数。 |
| `recommend_result_count` | Histogram | `source` | 单次推荐返回结果数量。 |
| `recommend_empty_total` | Counter | `source` | 推荐结果为空次数。 |
| `recommend_cache_hit_total` | Counter | `cache_type` | 缓存命中次数。 |
| `recommend_fallback_total` | Counter | `fallback_type` | 推荐兜底次数。 |
| `recommend_source_total` | Counter | `source` | 推荐结果来源次数。 |

推荐来源枚举：

```text
redis_realtime
redis_offline
final_csv
parallel_inference
popular_fallback
```

### 6.4 反馈业务指标

| 指标名 | 类型 | 标签 | 含义 |
| --- | --- | --- | --- |
| `feedback_events_total` | Counter | 无 | 反馈事件总数。 |
| `feedback_event_type_total` | Counter | `event_type` | 不同反馈类型数量。 |
| `recommend_exposure_total` | Counter | 无 | 推荐曝光次数。 |
| `recommend_click_total` | Counter | 无 | 点击次数。 |
| `recommend_collect_total` | Counter | 无 | 收藏或喜欢次数。 |
| `recommend_rating_total` | Counter | 无 | 评分次数。 |

| `model_inference_duration_seconds` | Histogram | `model_name` | 模型推理耗时。 |
| `model_inference_success_total` | Counter | `model_name` | 模型推理成功次数。 |
| `model_inference_error_total` | Counter | `model_name` | 模型推理失败次数。 |

## 7. 开发阶段安排

### 阶段一：监控基础模块

目标：建立 Prometheus 指标定义和统一导出能力。

任务：

- 引入 `prometheus-client`。
- 在 `monitor/metrics.py` 中定义统一 CollectorRegistry 或默认 registry。
- 保留现有 `build_prometheus_metrics` 的兼容输出。
- 将 `/metrics` 输出改为标准 Prometheus `generate_latest`。
- 为旧指标保留同名或兼容别名。

验收：

- 服务启动后访问 `/metrics` 成功。
- 原有 `reciperec_up`、`reciperec_uptime_seconds` 等指标仍可看到。
- 新增指标不会导致 `/metrics` 报错。

### 阶段二：系统资源采集

目标：采集机器和 FastAPI 进程资源指标。

任务：

- 引入 `psutil`。
- 新增 `monitor/system_metrics.py`。
- 实现 `collect_system_metrics()`，定时更新 Gauge。
- 在 `app/main.py` 的 lifespan 中启动后台采集线程或异步任务。
- 设置采集周期，建议默认 5 秒。
- 采集异常只写日志，不影响服务运行。

验收：

- `/metrics` 中可以看到 CPU、内存、磁盘、网络、进程资源指标。
- 采集线程异常不会导致 FastAPI 退出。
- 本地高频访问接口时，监控采集不会明显增加响应耗时。

### 阶段三：HTTP middleware

目标：自动统计所有 FastAPI 接口请求量、耗时、状态码和异常。

任务：

- 新增 `monitor/http_metrics.py`。
- 实现 HTTP middleware。
- 在请求开始时记录时间。
- 在响应返回时记录请求数、耗时和状态码。
- 在异常抛出时记录异常类型，并继续交给 FastAPI 原有异常处理。
- 在 `app/main.py` 中注册 middleware。

注意：

- 不捕获并吞掉业务异常。
- 不修改 response body。
- 不新增业务 header，除非确认前端不会依赖原 header 集合。
- 路径标签优先使用 `request.scope["route"].path`，避免真实 ID 进入标签。

验收：

- 调用 `/health`、`/recipes/recommend/{user_id}`、`/recipes/search` 后，对应 HTTP 指标增加。
- 访问不存在路径时，404 状态码能被统计。
- 人为触发 500 时，异常指标能增加。

### 阶段四：推荐业务指标

目标：在推荐接口中记录缓存命中、结果数量、空结果和兜底情况。

任务：

- 新增 `monitor/business_metrics.py`。
- 在 `api/routes.py` 的 `/recipes/recommend/{user_id}` 接口中增加旁路指标调用。
- 实时推荐缓存命中时记录 `redis_realtime`。
- 离线推荐缓存命中时记录 `redis_offline`。
- 离线 CSV 读取成功时记录 `final_csv`。
- 推荐结果为空时记录 `recommend_empty_total`。
- 后续接入多模型并发推理时记录 `parallel_inference`。
- 热门推荐兜底时记录 `popular_fallback`。

注意：

- 指标调用应放在已有业务分支之后，不改变原分支判断。
- 指标函数内部需要捕获异常，避免监控失败影响推荐返回。
- 推荐结果数量只记录数量，不记录具体菜谱 ID。

验收：

- 命中实时缓存、离线缓存、CSV 推荐时，`recommend_source_total` 和 `recommend_cache_hit_total` 正确变化。
- 返回空结果时，`recommend_empty_total` 增加。
- 推荐接口返回内容与改造前保持一致。

### 阶段五：反馈业务指标

目标：记录反馈、曝光、点击、喜欢、评分等事件数量。

任务：

- 在 `feedback/feedback_service.py` 中，仅在数据库写入成功后记录指标。
- `record_feedback` 成功后记录 `feedback_events_total` 和 `feedback_event_type_total`。
- 根据 `feedback_type` 分别更新点击、收藏、评分、负反馈等指标。
- `record_exposure` 成功后记录 `recommend_exposure_total`。

注意：

- 不把 `user_id`、`movie_id`、`recipe_id` 作为 Prometheus 标签。
- Kafka 发送失败不影响指标记录，指标只代表后端接收到并持久化成功的事件。

验收：

- 调用 `/feedback` 后，反馈指标增加。
- 调用 `/recommendation-exposure` 后，曝光指标增加。
- 反馈接口原响应结构保持不变。

### 阶段六：Prometheus 与 Grafana

目标：提供基础监控展示能力。

任务：

- 新增 `config/prometheus.yml`。
- 配置 Prometheus 抓取 FastAPI `/metrics`。
- 新增 Grafana 看板 JSON 或看板说明文档。
- 可选：在 `docker-compose.yml` 中增加 Prometheus 和 Grafana 服务。

建议：

- 如果担心影响现有 Docker 启动，优先提供单独的 `docker-compose.monitor.yml`。
- 主 `docker-compose.yml` 暂不强制加入监控服务，避免改变默认运行方式。

验收：

- Prometheus Targets 页面显示 FastAPI target 为 `UP`。
- Grafana 能展示 CPU、内存、接口 QPS、接口耗时、错误数、推荐来源、缓存命中、反馈事件。

### 阶段七：测试与文档

目标：保证监控接入稳定、可回退、可验收。

任务：

- 编写或补充监控模块单元测试。
- 编写 `/metrics` 接口 smoke test。
- 编写推荐接口改造前后返回结构对比说明。
- 编写 Prometheus/Grafana 启动说明。
- 编写验收清单。

验收：

- 现有测试通过。
- 新增测试通过。
- 监控关闭后，系统仍可正常启动和访问核心接口。

## 8. 推荐实现顺序

建议按以下顺序开发：

```text
1. 增加依赖 prometheus-client、psutil
2. 重构 monitor/metrics.py，保留现有指标兼容
3. 新增系统资源采集 monitor/system_metrics.py
4. 在 app/main.py 中启动资源采集任务
5. 新增 HTTP middleware monitor/http_metrics.py
6. 在 app/main.py 中注册 middleware
7. 新增业务指标封装 monitor/business_metrics.py
8. 给推荐接口增加缓存命中、来源、空结果指标
9. 给反馈服务增加反馈事件指标
10. 增加 Prometheus 配置
11. 增加 Grafana 看板或说明
12. 补充测试和验收文档
```

## 9. 不影响其他功能的具体措施

### 9.1 配置开关

建议增加以下配置项：

```text
MONITORING_ENABLED=true
MONITORING_SYSTEM_COLLECT_INTERVAL_SECONDS=5
MONITORING_HTTP_METRICS_ENABLED=true
MONITORING_BUSINESS_METRICS_ENABLED=true
```

当 `MONITORING_ENABLED=false` 时：

- 不启动系统资源采集线程。
- middleware 可以不注册，或注册后不记录指标。
- 业务指标函数直接 return。
- `/metrics` 至少保留 `reciperec_up`，方便判断服务存活。

### 9.2 指标函数安全封装

业务代码调用监控函数时，应使用安全封装：

```python
try:
    record_recommend_source("final_csv")
except Exception:
    logger.debug("record metric failed", exc_info=True)
```

也可以在 `business_metrics.py` 内部统一捕获异常，业务模块只调用函数。

### 9.3 避免高成本采集

- 系统资源指标定时采集，不在每个请求中实时采集。
- `/metrics` 不再频繁查询大表或加载大 CSV。
- 如需保留 SQLite 统计，建议降低查询频率并做内存缓存。

### 9.4 不修改业务主流程

推荐接口只允许增加类似以下逻辑：

```text
原业务分支判断
  -> 原业务结果
  -> 记录指标
  -> 返回原业务结果
```

不允许为了监控改变：

- 推荐排序。
- 推荐数量。
- 缓存 key。
- 兜底优先级。
- API 返回字段。
- HTTP 状态码。

## 11. 验收标准

必须满足：

- FastAPI 启动后可以访问 `/metrics`。
- `/metrics` 中可以看到系统资源、HTTP、推荐、反馈等指标。
- 调用任意接口后，HTTP 请求数和耗时指标会变化。
- 调用推荐接口后，推荐请求数、结果数量、来源指标会变化。
- 调用反馈接口后，反馈事件指标会变化。
- Prometheus 能成功抓取 FastAPI 指标。
- Grafana 能展示基础看板。
- 推荐、搜索、反馈、详情等原有接口返回结构不变。
- 关闭监控配置后，系统核心功能仍可正常运行。

## 12. 风险与规避

| 风险 | 影响 | 规避措施 |
| --- | --- | --- |
| `/metrics` 查询数据库过慢 | Prometheus 抓取超时 | 用内存指标替代频繁 SQL，必要统计做缓存。 |
| middleware 捕获异常方式不当 | 改变接口错误行为 | 只记录异常，再重新抛出。 |
| 标签包含用户 ID 或菜谱 ID | Prometheus 时间序列爆炸 | 标签白名单控制，只允许低基数字段。 |
| 采集线程异常 | 影响服务稳定性 | 后台任务内部捕获异常并继续下一轮。 |
| Docker 默认服务变重 | 影响原启动体验 | 监控服务放到可选 compose 文件。 |
| 新依赖未安装 | 服务启动失败 | 更新 `requirements.txt` 并在部署说明中写明安装步骤。 |

## 15. 总结

本次系统性能监控开发的核心思路是：监控逻辑作为旁路能力接入现有系统，只观察、不干预。优先保证 `/metrics`、系统资源指标、HTTP middleware 和推荐业务指标可用，再逐步补充反馈指标和多模型推理预留指标。

只要严格遵守“不修改业务返回、不改变推荐链路、不使用高基数标签、监控异常不影响主流程”的原则，本次开发可以在提升系统可观测性的同时，最大限度保持现有功能稳定。
