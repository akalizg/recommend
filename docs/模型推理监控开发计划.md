# 模型推理监控开发计划

## 1. 开发目标

本计划用于补充 RecipeRec 项目中“模型推理监控”部分的开发工作。

该部分作为系统性能监控的扩展能力，重点观察 MapReduce 并发推荐链路中各个模型的执行情况，包括模型调用次数、耗时、成功、失败、超时、候选结果数量，以及 Reduce 融合阶段的输入输出和耗时。

本计划不替代现有的 `系统性能监控开发计划.md`，而是作为独立补充。现有系统监控看板继续关注系统资源、HTTP 接口、推荐业务和反馈业务；模型推理监控建议使用单独 Grafana 看板展示，避免一个看板内容过多、验收重点混乱。

## 2. 开发原则

| 原则 | 说明 |
| --- | --- |
| 不影响推荐结果 | 监控只记录指标，不参与推荐排序、召回、融合和兜底逻辑。 |
| 不影响团队功能 | 保留团队成员已实现的 MapReduce 并发推理代码，不用监控代码覆盖业务代码。 |
| 指标异常不阻断业务 | 模型指标记录失败时只写 debug 日志，不向上抛出异常。 |
| 低基数标签 | 只允许使用 `model_name`、`status` 等低基数字段，不记录 `user_id`、`recipe_id`、`request_id`。 |
| 看板独立 | 模型推理监控独立成新 dashboard，不堆到系统性能监控看板中。 |
| 可渐进接入 | 先接入当前并发推理链路，后续如新增模型，只需复用统一记录函数。 |

## 3. 监控范围

### 3.1 本次需要监控

- 每个模型的调用次数。
- 每个模型的成功次数。
- 每个模型的失败次数。
- 每个模型的超时次数。
- 每个模型的累计推理耗时。
- 每个模型返回的候选结果数量。
- Reduce 融合累计耗时。
- Reduce 输入候选数量。
- Reduce 输出推荐数量。
- 推荐结果是否来自 `parallel_inference`。

### 3.2 本次不监控

- 不监控具体用户 ID。
- 不监控具体菜谱 ID。
- 不记录请求体、用户偏好文本或推荐理由全文。
- 不修改并发模型数量、超时时间和融合权重。
- 不重构 MapReduce 推荐算法。
- 不实现告警系统，只提供 Prometheus 指标和 Grafana 展示。

## 4. 指标设计

| 指标名 | 类型 | 标签 | 含义 |
| --- | --- | --- | --- |
| `model_inference_requests_total` | Counter | `model_name`, `status` | 每个模型按状态统计的调用次数。 |
| `model_inference_success_total` | Counter | `model_name` | 模型推理成功次数。 |
| `model_inference_error_total` | Counter | `model_name` | 模型推理异常次数。 |
| `model_inference_timeout_total` | Counter | `model_name` | 模型推理超时次数。 |
| `model_inference_duration_seconds_total` | Gauge/Counter | `model_name` | 模型累计推理耗时，单位秒。 |
| `model_inference_duration_seconds_count` | Gauge/Counter | `model_name` | 模型耗时观测次数，用于计算平均耗时。 |
| `model_inference_result_count` | Gauge/Counter | `model_name` | 每个模型累计返回的候选数量。 |
| `recommend_reduce_duration_seconds_total` | Gauge/Counter | 无 | Reduce 融合累计耗时，单位秒。 |
| `recommend_reduce_duration_seconds_count` | Gauge/Counter | 无 | Reduce 融合执行次数。 |
| `recommend_reduce_input_count` | Gauge/Counter | 无 | Reduce 输入候选总数。 |
| `recommend_reduce_output_count` | Gauge/Counter | 无 | Reduce 输出推荐总数。 |
| `recommend_source_total{source="parallel_inference"}` | Counter | `source` | 统计推荐结果是否来自并发推理链路。 |

## 6. Grafana 看板规划

建议新增独立看板：

```text
RecipeRec Model Inference Monitoring
```

不要把模型指标堆到现有：

```text
RecipeRec System Monitoring
```

### 6.1 看板分区

| 分区 | 面板 |
| --- | --- |
| 模型调用概览 | 模型调用次数、成功次数、失败次数、超时次数。 |
| 模型耗时 | 各模型平均耗时、累计耗时。 |
| 模型产出 | 各模型候选结果数量。 |
| Reduce 融合 | Reduce 平均耗时、输入候选数量、输出推荐数量。 |
| 推荐链路来源 | `parallel_inference`、`final_csv`、`redis_offline`、`popular_fallback` 来源对比。 |

### 6.2 推荐 PromQL

模型调用次数：

```promql
sum(increase(model_inference_requests_total[5m])) by (model_name, status)
```

模型平均耗时：

```promql
sum(increase(model_inference_duration_seconds_total[5m])) by (model_name)
/
sum(increase(model_inference_duration_seconds_count[5m])) by (model_name)
```

模型返回候选数量：

```promql
sum(increase(model_inference_result_count[5m])) by (model_name)
```

Reduce 平均耗时：

```promql
increase(recommend_reduce_duration_seconds_total[5m])
/
increase(recommend_reduce_duration_seconds_count[5m])
```

Reduce 输入输出数量：

```promql
increase(recommend_reduce_input_count[5m])
increase(recommend_reduce_output_count[5m])
```

并发推理来源：

```promql
sum(increase(recommend_source_total{source="parallel_inference"}[5m]))
```

## 7. 开发阶段安排

### 阶段一：确认当前并发推理链路

目标：明确模型推理链路的入口和返回结构。

任务：

- 确认 `/recommend/{user_id}` 在未命中缓存时是否会进入 `ParallelInferenceService`。
- 确认 `ParallelInferenceResponse.model_results` 是否包含模型名、状态、耗时、候选列表。
- 确认 `reduce_duration_ms` 是否可用于 Reduce 耗时指标。

验收：

- 能通过一次推荐请求拿到模型执行结果。
- 不改变推荐接口响应结构。

### 阶段二：接入模型指标记录

目标：把模型推理指标纳入现有 Prometheus 指标体系。

任务：

- 在 `monitor/business_metrics.py` 中增加模型指标定义。
- 增加 `record_model_inference()`。
- 增加 `record_reduce()`。
- 保证函数内部捕获异常，不影响推荐接口。

验收：

- `/metrics` 能看到 `model_inference_*` 指标。
- `/metrics` 能看到 `recommend_reduce_*` 指标。
- 指标函数异常不会影响推荐接口。

### 阶段三：在推荐链路中埋点

目标：在并发推理完成后记录模型和 Reduce 指标。

任务：

- 在 `_record_parallel_metrics()` 中调用 `record_model_inference()`。
- 在 Reduce 完成后调用 `record_reduce()`。
- 在推荐来源为并发推理时记录 `recommend_source_total{source="parallel_inference"}`。

验收：

- 调用 `/recommend/{user_id}` 后，模型指标变化。
- 有模型超时时，timeout 指标增加。
- 有模型异常时，error 指标增加。
- 推荐结果仍然正常返回。

### 阶段四：新增 Grafana 模型监控看板

目标：把模型指标从系统监控看板中拆出来，单独展示。

任务：

- 新增 `reciperec_model_inference.json`。
- 配置模型调用、耗时、失败、超时、Reduce 输入输出等面板。
- 修改 Grafana provisioning，让新看板自动加载。

验收：

- Grafana 中可以看到 `RecipeRec Model Inference Monitoring` 看板。
- 触发推荐请求后，模型面板出现数据。
- 原 `RecipeRec System Monitoring` 看板保持不拥挤。

### 阶段五：测试与文档

目标：保证模型监控可验收、可复现。

任务：

- 补充模型指标导出测试。
- 补充并发推理指标记录测试。
- 编写前端或接口触发说明。
- 补充 Grafana 看板验收步骤。

验收：

- 监控相关测试通过。
- 并发推理测试通过。
- 文档说明可以指导用户触发模型监控数据。

## 8. 前端触发方式

当前最稳定的触发方式是调用推荐接口：

```text
GET /recommend/{user_id}
```

如果希望完全通过前端页面触发，需要在前端增加一个明确入口，例如：

```text
智能推荐 -> 加载我的推荐
```

该按钮调用：

```text
GET /recommend/{user_id}?top_k=20&use_cache=true
```

这样可以稳定触发：

- `recommend_source_total`
- `model_inference_requests_total`
- `model_inference_duration_seconds_total`
- `model_inference_result_count`
- `recommend_reduce_duration_seconds_total`
- `recommend_reduce_input_count`
- `recommend_reduce_output_count`

## 9. 验收标准

必须满足：

- `/metrics` 可以看到模型推理指标。
- 调用推荐接口后，模型调用次数增加。
- 模型成功、失败、超时能够按状态区分。
- Reduce 输入、输出、耗时指标能够变化。
- Grafana 新看板能够展示模型指标。
- 现有系统监控看板不被模型指标堆满。
- 推荐接口响应结构不变。
- 推荐、搜索、反馈、饭搭子等现有功能不受影响。

## 10. 风险与规避

| 风险 | 影响 | 规避措施 |
| --- | --- | --- |
| 指标重复定义 | FastAPI 启动时报 Prometheus duplicate metric | 所有模型指标统一放在 `monitor/business_metrics.py`。 |
| 指标标签过多 | Prometheus 时间序列膨胀 | 只使用 `model_name`、`status`，不使用用户和菜谱维度。 |
| 监控逻辑影响推荐 | 推荐接口变慢或异常 | 指标函数内部捕获异常，且只记录简单数值。 |
| 看板过于拥挤 | 验收展示不清晰 | 单独新增模型推理监控看板。 |
| 和团队 MapReduce 代码冲突 | 覆盖团队功能 | 以远端业务逻辑为基础，只补充旁路监控。 |

## 11. 交付物

- 模型推理指标记录函数。
- Reduce 指标记录函数。
- 推荐链路模型埋点。
- 独立 Grafana 模型推理看板。
- 模型监控验收说明。
- 指标导出与并发推理测试。

## 12. 总结

模型推理监控是系统性能监控的自然扩展，尤其适合配合团队的 MapReduce 并发优化进行验收。

建议保持两个看板：

- `RecipeRec System Monitoring`：看系统资源、HTTP、推荐业务和反馈业务。
- `RecipeRec Model Inference Monitoring`：看模型耗时、成功失败、超时和 Reduce 融合。

这样既不会影响原有系统功能，也能让不同成员的开发成果在 Grafana 中清晰展示。
