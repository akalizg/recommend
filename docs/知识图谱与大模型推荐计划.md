# 知识图谱与大模型推荐计划

## 一、项目目标

本计划旨在在现有食谱推荐系统基础上，进一步引入**知识图谱**与**大模型能力**，构建一个更智能、更可解释、支持对话交互的推荐系统。

整体目标包括：

1. 构建食谱领域知识图谱，沉淀食谱、食材、标签、营养、场景等结构化知识。
2. 使用知识图谱增强召回与排序，提升推荐相关性与覆盖率。
3. 结合大模型理解用户自然语言需求，支持对话式推荐。
4. 利用知识图谱路径和大模型生成更自然的推荐理由。
5. 形成“对话输入 → 图谱理解 → 多路召回 → 排序重排 → 理由生成 → 前端展示”的完整链路。

## 三、阶段一：构建轻量级知识图谱

### 3.1 目标

先从轻量图谱开始，验证数据建模、Neo4j 连接、图谱构建和图查询流程。

### 3.2 图谱建模思路

建议优先构建以下实体和关系：

- **Recipe**：菜谱节点
- **Ingredient**：食材节点
- **Tag**：标签节点
- **User**：用户节点（可选，后续增强）
- **Nutrition**：营养信息（可作为属性先存储）

关系示例：

- `(Recipe)-[:CONTAINS]->(Ingredient)`
- `(Recipe)-[:HAS_TAG]->(Tag)`
- `(User)-[:PREFERS]->(Tag)`
- `(User)-[:LIKES]->(Recipe)`
- `(Recipe)-[:SIMILAR_TO]->(Recipe)`（后续可生成）

### 3.3 Neo4j 安装与配置

1. 下载 Neo4j Community Edition。
2. 启动 Neo4j 服务。
3. 访问浏览器 `http://localhost:7474`。
4. 配置用户名和密码。
5. 在项目中保存连接信息。

### 3.4 配置文件

`knowledge_graph/config.py`

```python
NEO4J_URI = "bolt://localhost:7687"
NEO4J_USER = "neo4j"
NEO4J_PASSWORD = "your_password"
```

### 3.5 图谱构建脚本

`knowledge_graph/graph_builder.py`

核心职责：

- 读取 `recipe_metadata.csv`
- 创建 `Recipe` 节点
- 创建 `Ingredient` 节点
- 创建 `Tag` 节点
- 建立菜谱与食材、标签之间的关系
- 支持清库重建

建议先用小样本验证，例如 1000 条菜谱，确认逻辑正确后再跑全量数据。

### 3.6 索引建议

为了提升 Neo4j 查询性能，建议创建索引：

```cypher
CREATE INDEX recipe_id_idx FOR (r:Recipe) ON (r.recipe_id);
CREATE INDEX ingredient_name_idx FOR (i:Ingredient) ON (i.name);
CREATE INDEX tag_name_idx FOR (t:Tag) ON (t.name);
```

### 3.7 验证方式

在 Neo4j Browser 中执行：

```cypher
MATCH (n) RETURN n LIMIT 25
```

如果能看到节点和关系，说明图谱构建成功。

## 四、阶段二：图谱增强召回

### 4.1 目标

在已有召回体系基础上加入图谱召回，让系统不仅依赖协同过滤和内容相似度，还能利用“食材关系”“标签关系”“营养约束”等结构化知识。

### 4.2 图谱召回模块

`recall/kg_recall.py`

可实现以下召回策略：

1. **按食材召回**
   - 输入：`chicken`、`egg`、`potato`
   - 输出：包含这些食材的高相关菜谱

2. **按标签召回**
   - 输入：`quick`、`dinner`、`healthy`
   - 输出：符合场景标签的菜谱

3. **按营养约束召回**
   - 输入：最大热量、最小蛋白质等
   - 输出：符合营养限制的菜谱

4. **混合召回**
   - 将食材、标签、营养等信号融合成统一召回分数

### 4.3 融合方式

建议把知识图谱召回作为新增召回通道，接入现有多路召回融合逻辑中：

- ALS
- ItemCF
- FAISS
- LightGCN
- Content
- Hot
- KG Recall

这样可以提升候选集覆盖面，尤其适合冷启动、食材驱动和场景驱动推荐。

## 五、阶段三：图谱特征工程与排序优化

### 5.1 目标

让知识图谱不只是用于召回，还能进入排序模型，成为可学习的特征信号。

### 5.2 图谱特征示例

可以设计以下特征：

- 用户偏好标签与菜谱标签匹配数
- 用户历史喜欢食材与当前菜谱食材重合数
- 菜谱之间的食材相似度
- 用户历史高分菜谱与当前候选菜谱的路径相似度
- 菜谱是否属于用户常见场景标签
- 菜谱与用户长期兴趣标签的图路径得分

### 5.3 特征工程模块

`rank/kg_feature_engineering.py`

职责包括：

- 读取候选集
- 调用 Neo4j 查询图谱特征
- 将图谱特征拼接到排序样本中
- 供 LightGBM / XGBoost 重新训练

### 5.4 排序优化

将图谱特征加入排序模型后，可以重新训练：

- XGBoost
- LightGBM
- Logistic Regression

预期收益：

- 提升场景理解能力
- 提升解释性
- 提升用户偏好与菜谱内容的匹配精度

## 六、阶段四：长尾效应专项评估日志

为验证知识图谱是否真正帮助系统挖掘长尾菜谱，本项目对融合召回池、排序候选集和最终 Top50 推荐结果进行了长尾专项评估。长尾定义采用 `movie_profile.csv` 中 `movie_popularity` 的 70% 分位阈值，即 `movie_popularity <= 6.4694` 的菜谱视为长尾菜谱。

### 6.1 评估对象

- `data/recall/merged_recall_candidates.csv`
- `data/rank/rank_candidates.csv`
- `data/rank/enhanced/ranked_top50_enhanced.csv`

### 6.2 评估结果

- **融合召回池**
  - 行数：`2,507,600`
  - 唯一菜谱数：`113,249`
  - 长尾菜谱占比：`32.14%`
  - 独立长尾菜谱数：`68,472`
  - 独立长尾菜谱覆盖率：`60.46%`

- **排序候选集**
  - 行数：`2,507,600`
  - 唯一菜谱数：`113,242`
  - 长尾菜谱占比：`32.14%`
  - 独立长尾菜谱数：`68,464`
  - 独立长尾菜谱覆盖率：`60.46%`

- **最终 Top50 推荐结果**
  - 行数：`1,253,800`
  - 唯一菜谱数：`85,107`
  - 长尾菜谱占比：`37.80%`
  - 独立长尾菜谱数：`52,425`
  - 独立长尾菜谱覆盖率：`61.60%`

- **覆盖率指标**
  - 融合召回池唯一菜谱覆盖率：`4.52%`
  - 排序候选集唯一菜谱覆盖率：`4.52%`
  - 最终 Top50 唯一菜谱覆盖率：`6.79%`

### 6.3 结论

评估结果表明，知识图谱在保持排序模型稳定的前提下，能够明显提升长尾菜谱在最终推荐结果中的占比。最终 Top50 推荐中的长尾占比从召回阶段的 `32.14%` 提升至 `37.80%`，同时唯一菜谱覆盖率也从 `4.52%` 提升到 `6.79%`。这说明知识图谱不仅增强了推荐系统的候选覆盖，也有效促进了“隐藏宝藏”菜谱的挖掘。

## 八、阶段四：对话式智能推荐

### 6.1 目标

用户可以直接用自然语言描述需求，例如：

- “推荐一些低卡高蛋白的鸡肉菜”
- “我想要 30 分钟内能做完的晚餐”
- “给我找几个适合减脂的快手菜”

系统先理解意图，再做图谱召回与排序。

### 6.2 对话推荐接口

`api/chat_routes.py`

流程建议如下：

1. 接收用户自然语言输入
2. 调用大模型解析意图
3. 提取结构化条件，例如：
   - 食材
   - 标签
   - 热量限制
   - 蛋白质要求
4. 调用知识图谱召回
5. 结合排序模型筛选结果
6. 生成推荐理由
7. 返回前端展示

### 6.3 大模型接入方式

可以使用以下方式之一：

- OpenAI 风格接口
- DeepSeek API
- 本地大模型服务
- 规则解析作为兜底方案

建议先做“规则解析 + 图谱召回”版本，再逐步接入大模型增强。

### 6.4 推荐理由生成

推荐理由可结合：

- 菜谱标签命中
- 食材命中
- 营养符合用户目标
- 菜谱与用户历史偏好相似
- 图谱路径解释

例如：

- “这道菜包含鸡肉和鸡蛋，符合你的高蛋白需求。”
- “这道菜属于快手晚餐场景，制作时间约 25 分钟。”
- “它与用户历史喜欢的低脂标签匹配度较高。”
