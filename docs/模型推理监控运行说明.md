# 模型推理监控运行说明

本文档用于验收 `RecipeRec Model Inference Monitoring` 看板和模型推理 Prometheus 指标。

模型推理监控只做旁路记录，不参与推荐排序、召回、融合和兜底逻辑；如果指标记录失败，只会写 debug 日志，不会中断推荐接口。

## 1. 启动前确认

确认后端已启动，并且 `/metrics` 可以访问：

```powershell
Invoke-WebRequest "http://127.0.0.1:8000/metrics" -UseBasicParsing
```

确认并发推理开关处于开启状态。默认配置在 `app/config.py` 中：

```text
parallel_inference_enabled = true
```

如果本地通过环境变量覆盖过配置，需要保证：

```powershell
$env:PARALLEL_INFERENCE_ENABLED="true"
```

## 2. 启动 Prometheus 和 Grafana

本地 Windows 环境可以使用项目脚本启动：

```powershell
.\scripts\start_local_monitoring.ps1
```

如果 Prometheus 不在系统 PATH 中，需要传入本地 `prometheus.exe` 路径：

```powershell
.\scripts\start_local_monitoring.ps1 -PrometheusExe "D:\prometheus\prometheus.exe"
```

启动后访问：

```text
Prometheus: http://localhost:9090
Grafana:    http://localhost:3001
账号密码:   admin / admin
```

Grafana 看板位置：

```text
Dashboards -> RecipeRec -> RecipeRec Model Inference Monitoring
```

## 3. 触发模型推理数据

推荐接口命中 Redis 缓存时，不会重新进入模型推理链路。验收模型推理监控时建议临时关闭本次请求的缓存读取：

```powershell
Invoke-RestMethod "http://127.0.0.1:8000/recommend/1535?top_k=10&use_cache=false"
```

多执行几次可以让 Grafana 曲线更明显：

```powershell
1..5 | ForEach-Object {
    Invoke-RestMethod "http://127.0.0.1:8000/recommend/1535?top_k=10&use_cache=false" | Out-Null
    Start-Sleep -Seconds 2
}
```

如果希望通过前端页面触发，操作路径是进入推荐页面并刷新或加载“我的推荐”。为了稳定触发模型推理，前端请求需要使用：

```text
GET /recommend/{user_id}?top_k=10&use_cache=false
```

如果前端默认使用 `use_cache=true`，已经存在缓存时只会触发推荐来源或缓存指标，不一定会触发 `model_inference_*` 和 `recommend_reduce_*` 指标。

## 4. Prometheus 验收查询

进入 Prometheus 的 Graph 页面，依次查询：

```promql
model_inference_requests_total
```

```promql
model_inference_success_total
```

```promql
model_inference_error_total
```

```promql
model_inference_timeout_total
```

```promql
model_inference_duration_seconds_total
```

```promql
model_inference_result_count
```

```promql
recommend_reduce_duration_seconds_total
```

```promql
recommend_reduce_input_count
```

```promql
recommend_reduce_output_count
```

```promql
recommend_source_total{source="parallel_inference"}
```

如果查询不到数据，先确认：

- 后端 `/metrics` 能打开。
- Prometheus `Status -> Targets` 中 `reciperec-api-local` 或 `reciperec-api` 是 `UP`。
- 已调用 `/recommend/{user_id}?use_cache=false`。
- 调用后等待至少一个 Prometheus 抓取周期，本地默认约 15 秒。

## 5. Grafana 验收看板

打开 `RecipeRec Model Inference Monitoring` 后重点检查：

- `Model Requests By Status`：按模型名和状态展示调用次数。
- `Model Average Latency`：展示各模型平均推理耗时。
- `Model Candidate Output`：展示各模型候选结果数量。
- `Model Success Error Timeout`：展示成功、异常和超时次数。
- `Reduce Average Latency`：展示 Reduce 融合平均耗时。
- `Reduce Input And Output`：展示 Reduce 输入候选数和输出推荐数。
- `Recommendation Source`：展示推荐来源分布。
- `Parallel Inference Source`：展示并发推理链路命中次数。

新看板独立于 `RecipeRec System Monitoring`，不会把模型指标堆到系统性能监控看板上。

## 6. 验收通过标准

- `/metrics` 中能看到 `model_inference_*` 指标。
- `/metrics` 中能看到 `recommend_reduce_*` 指标。
- 调用推荐接口后，模型调用次数、耗时和候选数量发生变化。
- 推荐来源出现 `parallel_inference`。
- Grafana 中能打开 `RecipeRec Model Inference Monitoring`。
- 原有推荐、搜索、反馈和系统监控功能保持正常。
