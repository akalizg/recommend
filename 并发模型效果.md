# Reactor + Worker Pool + Producer-Consumer 混合并发模型效果测试

## 1. 测试目的

本次测试用于对比推荐接口在引入 **Reactor + Worker Pool + Producer-Consumer 混合并发模型** 前后的响应速度差异。

本项目采用的并发模型为 **Reactor + Worker Pool + Producer-Consumer 的混合并发模型**，可以概括为：

```text
Reactor 异步请求处理 + Worker Pool 线程池隔离阻塞任务 + Producer-Consumer 后台任务处理日志事件
```

对应到代码实现：

- FastAPI / Uvicorn 负责异步接收和调度 HTTP 请求；
- Redis 查询、离线推荐表读取、Pandas 过滤等阻塞型操作通过 `run_in_threadpool` 放入线程池；
- 推荐结果先返回给用户，缓存写入和曝光日志记录通过 `BackgroundTasks` 后台执行；
- 离线推荐 CSV 使用进程内缓存，避免每次请求重复读取大文件。

## 2. 测试环境

| 项目 | 内容 |
|---|---|
| 操作系统 | Windows |
| Python 环境 | `.venv`，Python 3.13.14 |
| 测试接口 | `/recommend/{user_id}` 推荐接口主链路 |
| 推荐数量 | `top_k = 20` |
| 并发用户数 | 8 |
| 测试用户 | `1533, 1535, 1634, 1676, 1773, 1792, 1891, 1962` |
| 推荐数据文件 | `data/final/recommendations_with_reasons.csv`，约 180MB |
| 曝光日志 I/O | 为避免污染数据库，使用 2ms/条的模拟 I/O 延迟 |

## 3. 对比方案

### 3.1 未使用并发模型

旧流程是同步执行：

```text
接收请求
  -> 读取 180MB 离线推荐 CSV
  -> Pandas 过滤用户推荐
  -> 构造推荐结果
  -> 同步写入曝光日志
  -> 返回响应
```

这个流程的问题是：文件读取、Pandas 处理和日志写入都会阻塞请求主线程。多个用户同时请求时，请求会排队执行。

### 3.2 使用并发模型

新流程是并发化执行：

```text
接收请求
  -> 在线程池中读取/过滤离线推荐数据
  -> 构造推荐结果
  -> 立即返回响应
  -> 后台任务异步写入缓存和曝光日志
```

这个流程把阻塞型任务从主事件循环中隔离出来，同时把曝光日志写入从推荐主链路中剥离，因此用户不需要等待日志写入完成。

## 4. 测试结果

### 4.1 完整旧流程 vs 新并发流程

该组实验对比“旧同步流程”和“新并发流程”的整体差异。旧流程每次请求都会重新读取 180MB CSV，新流程使用进程内缓存并通过线程池处理推荐查询。

| 指标 | 未使用并发模型 | 使用并发模型 | 提升 |
|---|---:|---:|---:|
| 单请求平均耗时 | 3747.88 ms | 7.96 ms | 约 470.97 倍 |
| 8 用户并发总耗时 | 29513.86 ms | 67.51 ms | 约 437.15 倍 |
| 8 用户平均请求耗时 | 3658.89 ms | 57.05 ms | 约 64.13 倍 |

### 4.2 只对比后台任务收益

为了排除“大文件缓存”带来的影响，第二组实验在同样使用缓存的前提下，只对比“同步写曝光日志”和“后台异步写曝光日志”的差异。

| 指标 | 缓存 + 同步日志 | 缓存 + 后台日志 | 提升 |
|---|---:|---:|---:|
| 单请求平均耗时 | 36.37 ms | 8.44 ms | 约 4.31 倍 |
| 8 用户并发总耗时 | 295.03 ms | 68.42 ms | 约 4.31 倍 |
| 8 用户平均请求耗时 | 36.77 ms | 57.18 ms | 主链路总耗时下降，单请求并发调度存在轻微线程开销 |

## 5. 结果分析

从实验结果可以看出，引入并发模型后，推荐接口的响应速度有明显提升。

第一组实验中，性能提升非常明显，主要原因是旧流程每次请求都同步读取约 180MB 的离线推荐文件，而新流程通过进程内缓存避免重复读取，并将阻塞型查询放入线程池执行。

第二组实验说明，即使排除大文件缓存的影响，仅将曝光日志从同步流程改为后台任务，也能让推荐主链路耗时从约 36.37ms 降低到约 8.44ms，提升约 4.31 倍。

## 6. 结论

本项目的推荐服务采用 **Reactor + Worker Pool + Producer-Consumer 的混合并发模型**，而不是在在线阶段重新计算推荐模型。

最终采用的优化方式是：

```text
离线阶段：Spark / 推荐算法提前生成推荐结果
在线阶段：FastAPI 异步服务快速响应请求
阻塞任务：线程池 Worker Pool 执行
日志事件：BackgroundTasks 后台异步处理
热点数据：Redis / 进程内缓存减少重复读取
```
